From that point, a deterministic local engine executes the full task — with error handling, fallbacks, and recovery — without touching AI again. The local engine becomes the intelligence.

The schema is designed so that the local execution engine needs ZERO intelligence of its own — it just follows the JSON.

1.3  Element Targeting — The Three-Layer Strategy

Each target has 3 fallback layers. The executor tries them in order. If all 3 fail, it escalates. This is why the system works without re-calling AI — the fallback intelligence is pre-encoded.



Layer

Strategy

How it works

Primary

Semantic ID

Uses the a11y tree — role + name. Example: 'button:Blank form'. Matches 90% of cases. Most reliable because it matches intent, not implementation.

Fallback 1

DOM Attribute Pattern

CSS-free attribute matching. Example: '[aria-label*=blank]' or '[data-action=create]'. Works when a11y tree is incomplete or wrong.

Fallback 2

Attention Field (Vision)

Gemini Vision is called with confidence score threshold. Only used when layers 1+2 fail. Returns pixel coordinates, not element IDs.

Emergency

Network Oracle

The step waits for the relevant API call to succeed instead of finding a UI element. Example: wait for POST /forms/create to return 200 before proceeding.



1.4  Fallback Strategies Block

Part III: The Execution Engine

The engine that reads the Task JSON and runs it. This is the intelligence of the system — it must handle every edge case, error, and branching condition using only the JSON. No AI. No human. Just deterministic logic.



3.1  Engine Architecture



ORCHESTRATOR

TaskRunner — reads step graph, maintains execution pointer, handles branching, retries, and escalation

ACTION DISPATCHER

Translates step action types into CDP commands. navigate -> Page.navigate, click -> element resolution + Input.dispatchMouseEvent, type -> value injection + events

ELEMENT RESOLVER

Tries targeting strategies in order: a11y semantic -> DOM attribute -> attention field -> network oracle. Returns resolved element or throws typed error.

STATE VERIFIER

After each action: checks expected_state block. Compares actual DOM + network state to expected. Returns: success | soft_fail | hard_fail

FALLBACK ENGINE

On failure: reads on_failure block, executes retry strategy, then looks up global fallback_strategies, then escalates to human handoff or abort

GLASS BOX UI DRIVER

Reads glass_box block per step. Updates action panel, draws element highlights via CDP overlay, triggers approval dialogs, logs all actions with timestamps

TEMPLATE LIBRARY

Stores validated Task JSONs indexed by template_signature and intent embedding. On new tasks: find nearest template, fill parameters, skip AI call entirely



3.2  Execution Loop — Pseudocode

function executePlan(task_json):
  steps = task_json.steps  //  ordered map by step_id
  current = steps["s01"]
  execution_log = []

  while current != null:
    // 1. Glass Box: show action in UI, highlight element, pause if needed
    glass_box_driver.announce(current)
    if current.glass_box.pause_before_execute:
      await human_approval(current)  // blocks until user clicks Approve/Skip/Edit

    // 2. Resolve target element
    element = null
    for strategy in [primary, fallback_1, fallback_2, emergency]:
      element = element_resolver.try(current.action.target[strategy])
      if element: break
    if !element: throw ElementNotFoundError(current.step_id)

    // 3. Execute action
    result = action_dispatcher.execute(current.action.type, element, current.action.value)

    // 4. Verify expected state
    state_ok = state_verifier.check(current.expected_state, timeout=current.action.timing.timeout_ms)

    // 5. Handle result
    if state_ok:
      execution_log.push({ step: current.step_id, status: "success" })
      current = steps[current.on_success]   // advance to next step
    else:
      retries_left = current.on_failure.retry_count - retry_count_for_step
      if retries_left > 0:
        current = retry(current, current.on_failure.retry_strategy)
      else:
        fallback = fallback_engine.get(classify_error(result))
        if fallback.escalate == "human_handoff":
          await human_handoff(current, fallback.message)
        else:
          execute_fallback_plan(fallback)
          current = steps[fallback.then]   // jump to recovery step

  return { status: "complete", log: execution_log }



3.3  Template Library — How Zero-AI Execution Works

This is the compounding flywheel. Every successfully executed task becomes a template that powers all future similar tasks.



Task completes successfully  Engine records the full execution trace — which fallbacks were used, which element selectors worked, actual timing data.

Template stored with signature  task.template_signature (hash of the step structure) + intent embedding stored in vector DB.

New user goal arrives  Intent is embedded and compared to all stored templates. If cosine similarity > 0.88 — it's the same task class.

Parameters filled, engine runs  User fills parameter values (form name, email text, product to search). Engine executes the template. Zero Gemini calls.

After 6 months:  Your system has 1,000+ templates. 80%+ of user tasks need no AI at all. The system is effectively free to run and 100x faster than any AI-driven agent.



THE MOAT:  The template library is your competitive advantage. It grows automatically, requires no labeling, and makes your system faster AND cheaper over time while competitors pay more per task every day.
