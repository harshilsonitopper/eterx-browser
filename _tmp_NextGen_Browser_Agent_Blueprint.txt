2. Core Architecture

The agent is structured as 6 interconnected layers. Each layer has a single responsibility and communicates through well-defined interfaces. This separation lets you upgrade each layer independently.



Layer 6 — Orchestrator / Goal Manager

Accepts high-level user goals. Maintains task graph. Routes to Planner.

Layer 5 — Task Planner (Gemini)

Decomposes goals into ordered steps. Re-plans on failure. Uses chain-of-thought.

Layer 4 — Page Understanding

Fuses screenshot + a11y tree + DOM. Produces semantic page map.

Layer 3 — Action Engine

Executes clicks, types, scrolls, waits. Validates success after each action.

Layer 2 — Memory & State

Session memory, site knowledge base, task history. Persisted to disk.

Layer 1 — Chromium / CDP Interface

Raw browser control via Chrome DevTools Protocol. Tab management, network interception.



2.1  Layer 4: Page Understanding — The Key Unlock

WHY THIS MATTERS: This is the #1 reason current agents fail. They feed Gemini bad page representations. Fix this first and everything else gets easier.



You need to fuse 3 data sources into one unified page representation before calling Gemini:

Visual: Screenshot (visual context)

Use Chromium's Page.captureScreenshot via CDP

Annotate the screenshot: draw bounding boxes over interactive elements

Number each element so Gemini can reference them as [1], [2], [3]...

A11y Tree: Accessibility Tree (semantic structure)

CDP: Accessibility.getFullAXTree()

This gives you role, name, value for every element — far better than raw HTML

Filter out invisible/disabled nodes to reduce noise

DOM: DOM Snapshot (structural info)

Use DOM.getDocument() + CSS.getMatchedStylesForNode() for layout hints

Extract: tag, id, class, placeholder, aria-label, data attributes

Build a compressed element map: { id: 42, tag: 'input', label: 'Email', type: 'email' }



Gemini prompt structure for page understanding:

PROMPT TEMPLATE: Here is the page screenshot with numbered elements. Here is the element map: [...]. Here is the current task step: [STEP]. Which element should I interact with, and what action should I take? Respond with: {elementId, action, value, confidence}.



2.2  Layer 5: Task Planner — Hierarchical Decomposition

The planner takes a high-level goal and produces a deterministic task graph. Each node in the graph is an atomic, verifiable action.

Structure: Goal → Sub-goals → Steps → Atomic Actions (4-level hierarchy)

Schema: Each step has: description, success condition, rollback action, max retries

Verification: After each step completes, verify success before moving on

Recovery: On failure: re-plan with failure context fed back into Gemini



Example task graph for 'Create a Google Form':

TASK DECOMPOSITION: Goal → [Navigate to forms.google.com] → [Click '+ Blank'] → [Wait for editor to load] → [Click title field] → [Type form title] → [Add question] → [Set question type] → [Click Save]. Each step verified before proceeding.



2.3  Layer 3: Action Engine — Reliable Execution

The action engine wraps all CDP commands with retry logic, smart waiting, and post-action verification.

Clicking: Never click by CSS selector alone — use element center coordinates from bounding box

Typing: After typing, verify the value is actually in the field using DOM query

Navigation: Wait for network idle AND DOM stable before declaring a navigation complete

Dynamic content: Use MutationObserver via CDP to detect when dynamic content appears

Shadow DOM: Shadow DOM support: use CDP's pierce query mode for web components



2.4  Layer 2: Memory & State Management

Task State: Task State: Current step index, completed steps, failed steps, retry count

Session: Session Memory: Logged-in status per domain, cookies, auth tokens

Site KB: Site Knowledge Base: Known element patterns per domain (e.g. 'Google Forms new button is always class xyz')

Long-term: Long-term Memory: Past task outcomes stored in SQLite/vector DB for retrieval

4. Critical Fixes for Current Failures (Do These First)

Before starting the day plan, fix these 3 issues that are causing your current simple task failures:



FIX 1 — Stop using CSS selectors for clicking: Switch to: (1) get element bounding box from CDP, (2) click center coordinates using Input.dispatchMouseEvent. This alone fixes 60% of click failures on dynamic pages.



FIX 2 — Stop waiting fixed timeouts: Replace all sleep(2000) with: waitForNetworkIdle(500ms) + waitForDOMStable(300ms). Use CDP Page.lifecycleEvent to know exactly when the page is ready.



FIX 3 — Start using Accessibility Tree: Call Accessibility.getFullAXTree() every time before asking Gemini what to do. Feed it the role + name + value of each interactive element instead of raw HTML. Gemini will immediately make better decisions.



5. Gemini Prompting Strategy

5.1  Two-Model Architecture

Use two Gemini models with different roles for optimal speed and quality:

Flash: Gemini 2.0 Flash — Action selection. Fast, cheap. 'Given this page, what element do I click next?'

Pro: Gemini 2.5 Pro — Task planning and re-planning. Expensive, smart. 'Decompose this goal. Re-plan after this failure.'



5.2  Prompt Structure for Action Selection

SYSTEM: You are a browser automation agent. You see a numbered screenshot and an element manifest. Your job is to select the next action to complete the current step. Always respond with valid JSON only: {"elementId": number, "action": "click|type|scroll|wait", "value": "string or null", "confidence": 0-1, "reasoning": "brief explanation"}.



USER TURN: [annotated screenshot image] | Element manifest: [{id:1, role:'button', name:'Blank form', bbox:[x,y,w,h]}, ...] | Current step: 'Click the button to create a new blank Google Form' | Previous actions: [] | Last error: none



5.3  Few-Shot Examples to Include

Include 3-5 examples in your system prompt for common patterns:

How to identify a submit button even when it's labeled 'Next', 'Continue', or 'Done'

How to handle a dropdown — click to open, then click the option

How to fill a date picker — click field, type date, press Enter or click calendar day

How to detect and dismiss modal dialogs before continuing
