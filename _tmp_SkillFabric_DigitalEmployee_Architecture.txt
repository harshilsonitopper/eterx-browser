Part I: The 8 Core Subsystems

Skill Fabric is built from 8 tightly integrated subsystems. Each handles a distinct layer of autonomous operation. Together they produce a system that can take on any task, in any app, without pre-programming.



SUBSYSTEM 01  Skill Generation Engine

The system that makes the system unlimited

WHAT IT DOES

When the agent encounters a task it has no skill for, the Skill Generation Engine fires. It analyzes the task, the target app's interface patterns, and related existing skills — then synthesizes a new Skill Definition: a structured, parameterized, reusable capability. The new skill is stored permanently. Next time, it's used instantly with no generation needed. The agent's capability set grows without any developer involvement.

HOW TO BUILD IT

A Skill Definition is a structured JSON object: { skill_id, name, domain (app it targets), trigger_patterns (what user goals activate it), required_inputs (parameters), execution_plan (One-Shot JSON steps), success_criteria, related_skills }. Skill Generation prompt: feed Gemini the task goal + current app context + top-3 most similar existing skills + app knowledge profile. Output: complete Skill Definition JSON. Store in SkillVault (SQLite + vector index on name+trigger_patterns). Before generation: always query SkillVault — if similarity > 0.87, compose existing skills instead of generating new ones. Skill composition = combining 2+ skills into a workflow without AI calls.

WHY THIS IS NEW

Software tools are shipped with fixed APIs. No tool has ever been designed to generate its own new APIs at runtime from first principles. This borrows from program synthesis research (Gulwani, 2011) and meta-learning (MAML) — but applied to browser automation skills, not neural network weights. The closest concept is Wolfram's computational language, but applied to web UIs.



SUBSYSTEM 02  App Monitor Fabric

Always-on perception layer across every app the user cares about

WHAT IT DOES

The agent watches multiple apps simultaneously — Gmail, Slack, GitHub, Jira, Notion, dashboards — without the user having to ask. When something noteworthy happens (new email from VIP, PR merged, metric threshold crossed, deadline approaching), the Monitor Fabric triggers the right Skill automatically. It's not polling. It's not a cron job. It's a living, event-driven perception system that understands the semantic meaning of what it observes.

HOW TO BUILD IT

Each App Monitor is a persistent CDP browser context running in the background. Monitors use the Continuous Semantic Stream (SST) from prior architecture — real-time DOM event streaming, zero polling. Each monitor has a Monitor Rule: { app, watch_for (semantic condition), trigger_skill, priority, human_notify (boolean) }. Monitor Rules are generated by Skill Fabric from user-described needs: 'Tell me when any email mentions contract or invoice' -> auto-generates Monitor Rule + email-reading skill. Monitors run in low-CPU idle mode: MutationObserver active but sleeping. Wake on relevant DOM change. Classify change against all active Monitor Rules using lightweight local embedding match. Only call Gemini if classification confidence is below 0.8.

WHY THIS IS NEW

Background monitoring in current agents is entirely manual — developer writes a cron job, defines what to check, hardcodes the logic. App Monitor Fabric generates monitoring logic from natural language and runs it autonomously. This is the application of Complex Event Processing (CEP, used in financial trading systems) to browser-based app monitoring. No browser agent team has applied CEP theory to web app observation.



SUBSYSTEM 03  Autonomous Work Loop (AWL)

Self-sustaining task execution that runs indefinitely without human instruction

WHAT IT DOES

An Autonomous Work Loop is a persistent agent process assigned to a recurring responsibility — like an employee with a job description. You define the responsibility once in natural language ('Process all new support tickets: categorize them, draft a response, assign to the right team member, update the tracking sheet'). The AWL owns that responsibility permanently. It monitors for triggers, executes the full workflow when triggered, handles exceptions, and reports outcomes — without any further human instruction.

HOW TO BUILD IT

AWL Schema: { loop_id, responsibility_description, trigger (event from Monitor Fabric or schedule), skill_sequence (ordered list of Skill IDs to execute), exception_policy, reporting_config, human_escalation_threshold }. AWL Runner: persistent process that subscribes to Monitor Fabric events. On trigger: instantiate skill sequence with current context parameters. Execute using One-Shot JSON Brain execution engine. On exception: apply exception_policy (retry / skip / escalate). On completion: write to Work Log, send human summary if configured. Multiple AWLs run in parallel CDP contexts — each owns its own browser session so they never interfere. AWL Manager: global process that starts/stops/monitors all active loops, detects loops stuck in error state, restarts with backoff.

WHY THIS IS NEW

The concept of an autonomous agent with a 'job description' that executes indefinitely is not found in any current AI agent framework. Current agents are purely request-response. This borrows from Autonomous Agents in multi-agent systems research (Wooldridge & Jennings, 1995) and applies it to browser-based work execution. The difference from simple automation (Zapier, Make): AWLs are not static workflows — they adapt their skill sequence based on what they observe, using Skill Fabric to handle new situations.





SUBSYSTEM 04  Cross-App Skill Composer

Orchestrates skills across multiple apps as a single unified workflow

WHAT IT DOES

Most real work spans multiple apps. 'When a new lead comes in on HubSpot, research them on LinkedIn, check if we have history in Gmail, create a Notion brief, and schedule a follow-up in Google Calendar.' No existing agent can do this because each app is treated as an isolated silo. The Cross-App Skill Composer treats the entire set of apps a user works with as a single unified workspace and composes multi-app workflows from individual skills automatically.

HOW TO BUILD IT

App Graph: a directed graph where nodes are apps and edges represent data flows (output of one app becomes input of another). Built from observing the user's actual work patterns over time. Cross-App Workflow = a sequence of Skills from different app domains, stitched by data adapters that transform one app's output schema into another app's input schema. Data Adapters are themselves generated by Skill Fabric: 'convert HubSpot contact object to LinkedIn search query' is a micro-skill. Workflow Planner: given a multi-app goal, decompose into per-app skills, identify required data adapters, generate execution order based on dependencies (topological sort of the App Graph). Execute in parallel where steps are independent (no data dependency). CDP contexts per app run simultaneously.

WHY THIS IS NEW

Current automation tools (Zapier, Make, n8n) require manual pipeline construction with fixed connectors. Skill Fabric generates the connectors. The Cross-App Skill Composer applies dataflow programming (Kahn, 1974) — where computation is defined by data flowing between processing nodes — to browser-based app automation. The App Graph is a runtime-generated dataflow graph, not a manually wired pipeline.



SUBSYSTEM 05  Skill Performance Monitor & Auto-Optimizer

Every skill gets smarter after every execution — automatically

WHAT IT DOES

Skills are never static. Every time a skill executes, the result is scored against its success_criteria. Failures are analyzed. Slow steps are identified. Element selectors that failed are replaced with better ones. The skill's execution plan is quietly updated after each run. After 50 executions, a skill is dramatically more reliable than when it was first generated — without any developer touching it.

HOW TO BUILD IT

After each skill execution: record outcome (success/fail), per-step timings, which element targeting strategies were used, which fallbacks triggered. Skill Health Score = weighted average of: success_rate (60%), avg_execution_time (20%), fallback_rate (20%). Optimizer runs when Health Score drops below 0.75 OR when 3 consecutive failures occur. Optimizer prompt: feed Gemini the skill definition + last 5 execution traces + failure reasons. Output: updated skill definition with improved selectors, better timing parameters, new fallback strategies. Auto-patch the skill in SkillVault. Also run Skill Decay Detection: if a skill's success rate drops suddenly (site changed), flag for regeneration. Skill regeneration uses the original task intent + current page state to produce a fresh skill.

WHY THIS IS NEW

Self-optimizing code is the domain of genetic programming and automatic program repair (APR). Academic APR systems fix bugs in source code. This applies APR to browser automation skill definitions — treating execution traces as test cases and skill definitions as programs to be repaired. No automation platform has ever automatically repaired its own workflow definitions based on execution telemetry.



SUBSYSTEM 06  Productivity Intelligence Layer

The system that understands how you work and makes you work better

WHAT IT DOES

Beyond executing tasks, Skill Fabric observes how the user works and surfaces intelligence that makes them more productive: identifying where they spend most time, spotting repetitive manual tasks they haven't automated yet, detecting when they're in a meeting and pre-clearing their task queue, identifying which apps they switch between most and suggesting workflow integrations. It's not just doing work — it's actively improving how work gets done.

HOW TO BUILD IT

Work Pattern Graph: built from passively observing app usage (which apps, at what times, for how long, in what sequences). Stored as time-series vectors per app per hour of day. Pattern Miner runs weekly: identify top 10 repetitive action sequences that aren't yet automated. For each: estimate time cost per week, generate candidate skill, present to user as 'I noticed you do X manually every day — want me to automate it?'. Focus Mode Detector: classifies current work context (deep work, meeting, email triage, planning) from Monitor Fabric signals. Adjusts AWL behavior accordingly: suppress non-urgent notifications during deep work, run email triage AWL during transition periods. Weekly Productivity Report: generated autonomously from Work Pattern Graph — time spent per app, tasks automated vs manual, hours saved by Skill Fabric this week.

WHY THIS IS NEW

Productivity analytics tools (RescueTime, Toggl) observe and report. They never act on what they observe. Skill Fabric closes the loop: observe -> understand -> generate automation -> execute. This is the application of the observe-orient-decide-act (OODA) loop — originally a military decision framework — to personal productivity optimization.





SUBSYSTEM 07  SkillVault — The Living Capability Database

The growing brain of the system — every skill ever learned, forever

WHAT IT DOES

SkillVault is not a static library. It's a living, self-organizing knowledge base that stores every skill, indexes them for instant retrieval, detects redundancy, merges related skills, and evolves its own taxonomy over time. It is the persistent intelligence of the system — what makes today's Skill Fabric smarter than last month's without any retraining.

HOW TO BUILD IT

SkillVault schema: SQLite for skill records, FAISS/ChromaDB for vector similarity index on (name + trigger_patterns + domain). Skill retrieval: embed incoming task intent, query index for top-5 nearest skills, return with similarity scores. Skill Deduplication: weekly job that finds skill pairs with similarity > 0.92, merges them into a single generalized skill with the union of their parameters. Skill Taxonomy: auto-generated hierarchical categorization of all skills by domain, action type, app. New skills are auto-classified and placed in taxonomy. SkillVault Export: entire vault is exportable as a portable JSON package — share your skill library with another user of Skill Fabric. SkillVault Import: import another user's vault, deduplicate against your own, merge. Community skill sharing without any central server.

WHY THIS IS NEW

Vector databases for skills is similar to retrieval-augmented generation (RAG) for documents. But SkillVault is RAG for executable programs — retrieving not knowledge but capability. This has been explored academically as 'library-based program synthesis' but never shipped as a product. The community sharing model (export/import without central server) is borrowed from decentralized package managers like npm but applied to executable AI skills.



SUBSYSTEM 08  Human-in-the-Loop Escalation OS

Knows exactly when to work autonomously and when to involve a human

WHAT IT DOES

The hardest problem in autonomous systems is knowing the boundary of your own competence. Skill Fabric solves this with a multi-signal escalation system: confidence scoring on every decision, novelty detection (is this situation genuinely new?), risk classification on every action, and a structured human handoff protocol that gives the human full context and resumes autonomously the moment they respond. The result: maximum autonomy with zero unwanted surprises.

HOW TO BUILD IT

Escalation Score = weighted sum of: (1-skill_confidence) * 0.4 + action_risk_level * 0.3 + novelty_score * 0.2 + user_preference_weight * 0.1. Novelty Score: how different is the current execution context from the 10 most recent successful executions of this skill? Measured as cosine distance of current PageStateHash from historical successful states. Escalation Modes: Silent (score < 0.3, execute with log), Notify (0.3-0.6, execute but show notification), Preview (0.6-0.8, show what agent will do, 5s to cancel), Approval (0.8-0.9, require explicit confirm), Full Stop (>0.9, hand off with full context). Human Handoff Protocol: structured message with: what the agent was doing, what decision it needs help with, the options it sees, and a suggested default. One-tap response from human, agent resumes in under 2 seconds.

WHY THIS IS NEW

Current agents either always ask humans (slow) or never ask (dangerous). The idea of a dynamic, continuous competence boundary — an agent that knows its own confidence in real time and adjusts human involvement accordingly — is borrowed from active learning in ML (query by committee) and applied to autonomous task execution. No production agent system implements a mathematically grounded, multi-signal escalation score.

4.2  AWL Manager — System Design

Loop Registry:  All active AWL definitions stored in SQLite. AWL Manager reads on startup and instantiates runners for each.

CDP Context Pool:  Each AWL gets a dedicated CDP browser context. Contexts are pre-warmed (site already loaded, auth maintained). Zero cold-start penalty.

Event Bus:  Monitor Fabric publishes events. AWL Manager subscribes. On matching trigger: dequeue next available runner for that loop, execute skill sequence.

Execution Queue:  Each AWL has its own FIFO queue. Triggers arriving faster than execution speed are buffered up to the configured limit. Prevents cascade failures.

Health Dashboard:  Real-time view of all active loops: status (running/idle/error), executions today, success rate, last execution timestamp, queue depth.

Human Control:  Any loop can be paused, stopped, or reconfigured from the dashboard without restarting the system. Changes take effect on next trigger.

How all 8 subsystems connect. Data flows are shown top to bottom — perception feeds intelligence, intelligence feeds execution, execution feeds memory.



INPUT LAYER

All inputs arrive here and are classified, embedded, and routed

SKILL RESOLUTION LAYER

SkillVault Query -> Skill Match? -> Compose existing skills OR generate new skill via Skill Generation Engine

Zero AI calls for known tasks. One AI call for new tasks. Grows smarter with every task.

PLANNING LAYER

Cross-App Skill Composer -> App Graph -> Topological execution order -> Data adapter generation -> Parallel/sequential plan

Multi-app workflows assembled automatically from individual skills

MONITORING LAYER

App Monitor Fabric: persistent CDP contexts -> SST event streams -> Monitor Rule matching -> AWL trigger events

Always-on perception across all user apps. Event-driven, not polling.

EXECUTION LAYER

AWL Runners (parallel CDP contexts) -> One-Shot JSON Brain executor -> Glass Box UI -> Human-in-Loop escalation

Deterministic execution. Multiple workflows in parallel. Human control at every risk boundary.

OPTIMIZATION LAYER

Skill Performance Monitor -> execution telemetry -> health scoring -> auto-repair -> SkillVault update

Every skill gets better after every execution. No developer involvement.

INTELLIGENCE LAYER

Productivity Intelligence -> Work Pattern Graph -> automation suggestions -> Focus Mode -> Weekly Productivity Report

Observes how you work. Suggests and builds improvements. Closes the observe->automate loop.

MEMORY LAYER

SkillVault (executable capabilities) + App Knowledge Profiles + Work Pattern Graph + AWL execution history

Permanent, growing, self-organizing intelligence. Exportable and shareable.



How This Embeds Into Your Existing Browser Agent

Skill Generation Engine:  Wraps your existing Gemini API call. When your agent can't handle a task, Skill Fabric intercepts, generates a skill, and hands the execution plan back to your agent's executor.

App Monitor Fabric:  Runs as additional CDP contexts alongside your existing browser. Your agent's CDP connection is unaffected. Monitors are background threads.

AWL Runners:  Use your existing One-Shot JSON Brain executor as the execution engine. AWLs are just scheduled, looping callers of the same executor you already built.

SkillVault:  Drop-in replacement for your template library. Same interface, but with semantic search, composability graph, and auto-optimization on top.

Glass Box Safety:  Your existing risk classification and approval system — AWLs use the same Glass Box protocol for all escalations.
